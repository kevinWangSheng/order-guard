# Test & Verification Skill

当用户调用 `/verify` 或在完成一个任务需要验收时，按照本标准执行验证流程。

## 核心原则

- **零容忍**: 任何测试失败都必须修复，不允许跳过或忽略
- **全量运行**: 必须运行完整测试套件，不允许只跑部分测试
- **回归保护**: 新代码不能破坏已有测试
- **验收驱动**: 除了自动化测试，还要验证 specs/ 中的验收标准
- **流程完整**: 大任务必须验证从输入到输出的完整链路

## 任务规模判定

在开始验证前，先判定当前任务的规模：

**小任务**（单组件修改）：只改了一个文件/模块，如修 bug、改配置、调格式
→ 执行 Phase 1-3

**大任务**（跨层修改或新功能）：涉及多个模块、新增功能、架构变更
→ 执行 Phase 1-5（含全流程端到端验证）

判定依据：读取当前任务的 spec，如果涉及 2 个以上架构层或模块，则为大任务。

---

## Phase 1: 自动化测试（所有任务必须）

运行项目完整测试套件。

**判定标准:**
- 测试命令退出码必须为 0
- 不允许有 FAILED 或 ERROR 状态的测试
- 允许 SKIPPED（但需要检查 skip 原因是否合理，不能是为了逃避测试）
- 如果有测试失败，**立即停止后续步骤**，输出失败信息并修复

---

## Phase 2: 任务验收标准检查（所有任务必须）

1. 读取当前任务对应的验收标准文档（如 `specs/{task}/eval.md`）
2. 逐条检查验收标准：
   - 自动化测试能覆盖的：确认对应测试存在且通过
   - 需要手动验证的：执行验收文档中的命令并检查输出
3. 所有验收条目必须逐一确认，不允许跳过

---

## Phase 3: 冒烟测试（所有任务必须）

验证项目的所有入口点（如 Web 服务、CLI、模块导入）能正常启动，不会崩溃。

**标准:**
- 项目核心模块可正常导入
- 所有对外入口命令可正常执行（如 `--help`、`status` 等基础命令）
- 不要求业务逻辑正确，只要求不崩溃

---

## Phase 4: 全流程端到端验证（仅大任务）

按照项目的架构分层，从底层到顶层逐层验证完整的数据/业务流转管道。

**标准:**
- 识别项目的架构层次（如存储→配置→业务逻辑→对外接口）
- 每一层独立验证其核心功能可用
- 验证层与层之间的数据流转是否正常
- 如果有 E2E 测试，必须运行
- 如果支持 dry-run 模式，必须执行一次完整的 dry-run 走通全链路
- 缺少外部依赖（如 API Key、外部服务）时，标注 SKIPPED 并说明原因，不算失败

---

## Phase 5: 回归保护验证（仅大任务）

确保修改没有破坏已完成的功能。

**标准:**
- 检查项目中所有已标记为通过的功能/任务
- 确认它们对应的测试仍然全部通过
- 如果任何已通过功能的测试现在失败了 → **回归 Bug**，必须修复后才能继续

---

## 验证报告

所有 Phase 完成后，输出格式化报告：

```
## 验证报告

### 任务信息
- 任务ID: {task_id}
- 任务规模: 小任务 / 大任务

### Phase 1: 自动化测试
- 通过: X | 失败: X | 跳过: X
- 状态: PASS / FAIL

### Phase 2: 验收标准
- [x/✗] 逐条列出
- 状态: PASS / FAIL

### Phase 3: 冒烟测试
- [x/✗] 逐条列出
- 状态: PASS / FAIL

### Phase 4: 全流程验证（大任务才有）
- [x/✗/SKIPPED] 逐层列出
- 状态: PASS / FAIL

### Phase 5: 回归保护（大任务才有）
- 回归失败数: X
- 状态: PASS / FAIL

### 最终判定: PASS / FAIL
```

---

## 状态更新规则

**仅当所有执行的 Phase 都 PASS 时：**
1. 更新项目的任务跟踪文件，标记对应任务为通过
2. 告知用户验收通过

**任何 Phase 有 FAIL：**
1. **不更新任何状态文件**
2. 列出所有失败项及其所在 Phase
3. 提供修复建议
4. 等待用户指示是否修复

---

## 禁止行为

- NEVER 在测试失败时标记任务为完成
- NEVER 修改验收标准的内容来让验收通过
- NEVER 删除或跳过已有的测试用例
- NEVER 只运行部分测试来"通过验证"
- NEVER 添加 skip 标记来跳过失败的测试
- NEVER 修改断言来让失败的测试通过（除非断言本身有 bug）
- NEVER 跳过全流程验证来加速大任务的验收
- NEVER 忽略回归测试中的失败
